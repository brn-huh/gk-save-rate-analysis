## Problem Statement

넥슨 EA SPORTS FC 온라인 Open API로 매치 통계를 대량 수집·저장하고, 그 데이터로
**"어떤 골키퍼 카드가, 어떤 강화 단계(spGrade)에서 더 좋은가"를 판단**한다.
1차 지표는 GK 선방률(save rate)이며, 측정 모드는 **공식경기(matchtype=50)로 고정**한다.

> 핵심 난점: 공식경기는 사람이 직접 키핑하므로 raw 선방률은 카드 성능이 아니라
> "유저 손기술 + 상대 슛 난이도"에 크게 오염된다. 따라서 이 프로젝트의 본체는
> 수집이 아니라 **교란 보정 분석**이다.

## What Makes This Cool

- 게임 클라이언트나 커뮤니티 통계 사이트가 안 주는 질문에 답한다:
  "이 키퍼, 강화 올리면 진짜 덜 먹히나?" 를 데이터로.
- 실제 축구 분석의 **xG / Goals Saved Above expected(GSAx)** 기법을 게임 데이터에 이식.
  쉬운 슛만 받아 선방률 높은 카드와, 어려운 슛을 막아낸 카드를 구분한다.
- 결과물이 "선수 추천 근거"가 된다 — BP/현질 쓰기 전에 데이터로 검증.

## API 데이터 사전 (전 엔드포인트 실측, 2026-06-29 스펙 기준)

서버: `https://open.api.nexon.com` · 인증: 헤더 `x-nxopen-api-key`
데이터는 매시 정각 갱신, **2시간 지연**. 응답에 429(Too Many Requests) 정의됨 → 레이트리밋 존재.

문서 페이지 ↔ 카테고리 매핑: `?id=2`=User, `id=3`=Match, `id=4`=Ranker, `id=5`=Metadata, `id=6`=Image.
(실제 스펙 yaml: `static/api/fconline/{2,4,5,6}_script00000000000000.yaml`, Match는 `3_ko_script*.yaml`)

### User (id=2)
| 엔드포인트 | 입력 | 출력 필드 |
|---|---|---|
| `GET /fconline/v1/id` | nickname | `ouid` |
| `GET /fconline/v1/user/basic` | ouid | `ouid, nickname, level` |
| `GET /fconline/v1/user/maxdivision` | ouid | `matchType, division, achievementDate` |
| `GET /fconline/v1/user/match` | ouid, matchtype, offset, limit≤100 | 매치ID 배열 (최신순 desc) |
| `GET /fconline/v1/user/trade` | tradetype(buy/sell), offset, limit≤100 | `tradeDate, saleSn, spid, grade, value(BP)` (본인만) |

### Match (id=3) — ⭐ GK 데이터의 유일한 원천
| 엔드포인트 | 입력 | 출력 |
|---|---|---|
| `GET /fconline/v1/match` | matchtype, offset, limit≤100, orderby | **전체 매치ID 배열** (유저 무관, 최신순) |
| `GET /fconline/v1/match-detail` | matchid | 아래 MatchDetail 구조 |

**MatchDetail 구조 (요약):**
```
matchId, matchDate(UTC0), matchType
matchInfo[]  (참여 유저별, 보통 2명)
 ├ ouid, nickname, division
 ├ matchDetail{ seasonId, matchResult(승/무/패), matchEndType(0정상/1몰수승/2몰수패),
 │              systemPause, foul, injury, redCards, yellowCards, dribble(yd),
 │              cornerKick, possession(점유율), OffsideCount, averageRating, controller }
 ├ shoot{ shootTotal, effectiveShootTotal, shootOutScore,
 │        goalTotal(=실제 골), goalTotalDisplay, ownGoal,
 │        shootHeading, goalHeading, shootFreekick, goalFreekick,
 │        shootInPenalty, goalInPenalty, shootOutPenalty, goalOutPenalty,
 │        shootPenaltyKick, goalPenaltyKick }
 ├ shootDetail[]  ⭐ 슛 1개당: { goalTime, x, y, type(1~12), result(1유효/2빗나감/3골),
 │                 spId(슈터), spGrade, spLevel, spIdType(임대여부),
 │                 assist, assistSpI, assistX, assistY, hitPost, inPenalty }
 ├ pass{ passTry, passSuccess, short/long/bouncingLob/drivenGround/through/lobbedThrough … }
 ├ defence{ blockTry, blockSuccess, tackleTry, tackleSuccess }
 └ player[]  경기 사용 선수: { spId, spPosition, spGrade,
                status{ shoot, effectiveShoot, assist, goal, dribble, intercept,
                        defending, passTry, passSuccess, dribbleTry, dribbleSuccess,

```

> 선방률은 player 단독으로 안 나오고, 같은 경기 **상대팀 shoot/shootDetail**과 교차해야 나온다.
> ✅ player[].spGrade 가 존재 → "강화 몇 강" 분석은 match-detail로 가능 (랭커 API로는 불가).

## Premises (합의됨)

1. 측정 모드는 **공식경기(50)** 단일. 감독모드(52, AI 플레이로 교란이 적음)는 채택하지 않음.
   → 유저 손기술 교란을 **수집이 아니라 분석으로** 제거하기로 결정.
2. 목표 표본: **고유 매치 1만 건**에서 출발 (확장 가능).
3. GK 선방률은 1차 지표일 뿐, 최종 판단값은 **교란 보정 후 카드 효과**다.
4. spGrade 효과는 같은 카드(spId) 내부에서만 비교한다.

## 수집 파이프라인 (수정본)

원래 계획의 1단계(랭커 API로 ouid 대량 수집)는 **API상 불가능** → 두 경로 제시.

### 경로 A (권장, "최대한 많은 경기") — ouid 우회 생략
1. `meta/*.json` 5종 선다운로드 (matchtype로 50 확정, spposition로 GK=0 확정, spid·division 매핑 캐시).
2. `GET /v1/match?matchtype=50&offset=N&limit=100` 페이지네이션으로 매치ID 직접 수집.
   1만 건 = 약 100회 호출. (최대 offset 한계는 실측 확인 필요 — Open Question)
3. 매치ID별 `GET /v1/match-detail` 호출 → 원본 JSON 저장.
4. **슛 단위 테이블로 정규화** (아래 스키마). 여기서 GK↔상대팀 교차 join 수행.
5. spid/division/matchtype 메타 매핑.

### 경로 B (특정 표본만 필요할 때) — 유저 기반
1. 외부에서 닉네임 확보(공식 랭킹/커뮤니티) → `GET /v1/id`로 ouid.
2. ouid별 `GET /v1/user/match?matchtype=50` → 매치ID, Set으로 중복 제거하며 1만 건까지.
3. 이후 3~5단계는 경로 A와 동일.

> division은 match-detail에 들어오므로, 수집 단계에서 실력 필터를 안 걸고 **사후 필터**해도 된다.
> 따라서 "최대한 많은 경기"가 목표면 경로 A가 단순하고 빠르다.

### 운영 주의
- 429 레이트리밋 → 지수 백오프 + 동시성 제한. (정확한 한도는 발급 키 콘솔에서 확인)
- 데이터 2시간 지연 / 매시 갱신.
- 매치 통계 생성 전 상대가 종료하면 상대 정보가 비어있을 수 있음 → 교차 join 시 결측 처리 필수.

## 슛 단위 테이블 스키마 (분석의 기본 단위)

match-detail 1건 → 상대팀 `shootDetail[]`의 각 슛을 "내 GK가 받은 슛" 한 행으로 펼친다.
GK는 `matchInfo[나].player[]`에서 `spPosition==0` 인 엔트리로 식별.

```
shot_id, match_id, match_date, division, matchtype
gk_spid, gk_spgrade, gk_ouid            # 슛을 받은 키퍼 (내 팀)
shooter_spid, shooter_spgrade           # 슈터 (상대 팀)
x, y, dist_to_goal, angle_to_goal       # 슛 origin → 거리/각도 파생
shoot_type(1~12), in_penalty(bool)
result(1유효세이브 / 2빗나감 / 3골)
is_on_target = result in {1,3}
is_goal      = result == 3
```
선방률 1차식: `save_rate = #(result==1) / #(result in {1,3})` (유효슛 기준, 빗나간 슛 제외).

## 분석 설계 — "더 맞게" 만드는 4단계

raw 선방률은 기준선으로만. 위로 갈수록 정확하고 구현 난도도 올라간다.

- **Level 0 — raw 선방률.** 교란 그대로. 비교 기준선.
- **Level 1 — 난이도 보정 GSAx (필수).**
  전체 유효슛으로 로지스틱 xG 학습: `P(goal | dist, angle, shoot_type, in_penalty)`.
  GK별 `xGA = Σ P(goal)`, `GSAx = xGA − 실제실점`. 양수 = 기대보다 더 막음.
  → 슛 난이도 교란 제거. 쉬운 슛 받은 카드 거품 제거.
- **Level 2 — 유저 실력 보정 (유저 내 비교).**
  같은 ouid가 서로 다른 GK 카드를 쓴 경기들의 GSAx 차이만 집계.
  유저 손기술·전술·매칭 상대가 고정 → 카드 차이가 깨끗이 남음.
- **Level 3 — 혼합효과 모델 (목표).**
  슛 단위 로지스틱 혼합모델:
  `P(goal|유효슛) ~ dist + angle + shoot_type + in_penalty + division
   + (1|ouid) + (1|gk_spid) + (1|gk_spid:gk_spgrade)`
  → `(1|gk_spid)` 랜덤효과 = 난이도·유저·등급 다 뺀 **카드 순수 실점억제력 = 최종 순위값**.
  → `(1|gk_spid:gk_spgrade)` = 강화 효과. 표본 적은 칸은 자동 수축(shrinkage).

### 강화(spGrade) 효과
같은 spId 내부에서만 spGrade 비교(다른 카드끼리 비교 금지 — 카드효과와 섞임).
보조: 동일 카드·동일 유저가 강화를 올린 전후 GSAx 변화.

### 표본·순위 거품 방지
(gk_spid × gk_spgrade) 칸은 유효슛 N개(예: ≥30) 이상만 순위에 노출.
Level 3 미구현 시엔 Empirical-Bayes(Beta-Binomial) 수축으로 평균 회귀.

## 정직한 한계 (설계에 명시)

- **골문 내 위치 데이터 없음** → PSxG(슛이 골 어디로 갔는지) 불가. 반사신경 vs 운 완전 분리 불가.
  슛 origin 기반 xG가 상한.
- **수비수도 카드** → GK 앞 수비가 슛 질을 바꿈. GSAx가 일부 완화하나 완전치 않음.
- **선택편향** → 측정값은 "그 카드를 고른 유저 집단에서의 효과". 아무도 안 쓰는 카드는 추정 불가.
- **공식경기 한정** → 유저 손기술이 최대 교란. Level 2/3로 흡수하지만 잔여 교란 존재.

## What I noticed about how you think

- "분석을 더해서 더 맞게 하는 게 좋을 것 같아" — 편한 raw 선방률로 끝내지 않고
  교란 보정을 *프로젝트의 본체*로 받아들였다. 데이터를 의심할 줄 아는 태도.
- 직접 4단계 수집 순서를 짜와서 "참고해봐"라고 했다. 남이 떠먹여주길 기다리지 않고
  먼저 가설을 세워 검증받는 방식.
- "공식경기로만 측정" 처럼 측정 조건을 스스로 고정했다 — 변수를 줄여야 비교가 산다는 걸 안다.
- "다시 처음부터 이 문서를 기준으로 플랜을 세우겠다" — 탐색과 실행을 분리하는 빌더의 작업 순서.
